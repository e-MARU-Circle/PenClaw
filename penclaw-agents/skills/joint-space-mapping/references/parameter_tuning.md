# パラメータ調整・注意

## 距離の意味（再確認）

`vtkDistancePolyDataFilter` の**符号なし距離**。source の各点から target 表面への最近接距離[mm]。
内外を区別しない（貫通していても正の距離になる）。クリアランス・ギャップの「大きさ」を見る用途。

元アプリで確定している高速化設定をそのまま使っている:
- `SignedDistanceOff()`
- `ComputeSecondDistanceOff()` / `ComputeCellCenterDistanceOff()` → 逆方向・セル中心の距離は
  計算しない。**約24倍高速で精度は同一**（v2.1.0 CHANGELOG の実測）。

## デシメーション（速度 vs 精度）

| 設定 | 速度 | 精度 | 使いどころ |
|---|---|---|---|
| `--reduction` 無し | 遅い | 最高 | 小さいメッシュ・最終出力 |
| `--source-only-decimate --reduction 0.5` | 速い | 高い | **推奨の既定**。色が乗る source だけ間引き、距離参照先の target は原精度を保つ |
| `--reduction 0.5`（両方間引き） | 最速 | 中 | プレビュー・大規模メッシュの下見 |

`--reduction` は 0.0–0.95。0.5 = 三角形を約半分に間引く。0.9 を超えると形状が崩れて距離が荒れる。

## 部位限定で高速化 — 自動ROI `--roi-near-target` が既定の推奨

最も効くのは「計算する範囲そのものを減らす」こと。ROI は2系統あり、**通常は自動の
`--roi-near-target` を使う**。手動 `--roi-sphere` は座標が分かっている研究・バッチ向け。

### `--roi-near-target [MARGIN_MM]`（自動・術者は座標入力不要）

「関節スペース＝2面が近接している部位」という定義そのものを使い、**target（関節窩）表面から
MARGIN mm 以内の source 点だけ**を自動抽出してから距離を計算する。術者の指定は「関節窩から
何mm以内を見るか」のマージン1個だけ（フラグのみなら既定10mm）。座標は一切要らない。

- 内部は2段。①target のバウンディングボックス + margin で粗くマスク（numpy・高速）→
  ②残りを target 表面距離 ≤ margin で精密化（`vtkImplicitPolyDataDistance`）。粗マスクで点を
  激減させてから精密評価するので速い。
- **前提**: target が関節窩部位のメッシュになっていること（全顎ではなく窩部）。target が
  広いと ROI も広がる。target 側を部位限定しておくほどキレる。
- source 側だけを絞るので、残した部位の距離値は不変。
- margin の目安: 関節スペースの上限（TMJスケールの 5mm）+ 余裕で 8–12mm 程度。狭すぎると
  関節面の縁が欠け、広すぎると速度メリットが減る。
- 距離プリパス（粗距離→近接部のみ再計算）方式は未実装。必要なら追加可。

### `--roi-zmin/--roi-zmax`（X・Yも）軸平行ボックス — 下顎頭抽出の定番

各軸の下限/上限を任意に指定して source を箱で切り出す（指定した軸条件だけ効く）。**全顎の下顎
メッシュから下顎頭だけを抜く**のに最適で、`--roi-zmin 80` の1個で済む（Z上方＝下顎頭の座標系の
場合）。numpy の bbox マスクなので高速。

実運用の勘所:
- 全顎をそのまま全体距離にかけると、下顎頭の関節面以外（頸部・体部）が 5mm 超で青に振り切れ、
  下顎頭が暗く埋もれて見える。`--roi-zmin` で下顎頭領域に絞ると source 点が激減し、**間引きなしの
  原精度でも数秒**で計算でき、関節面の関節スペースが鮮明になる。
- 実測（CBCT 全顎・下顎 45万点）: `--roi-zmin 80` → 35,045点、原精度で約4秒、距離 min 1.0 /
  max 17.7 / mean 7.9mm、関節面が 1〜5mm で色づく。
- Z 下限の値は症例の座標系に依存する。一度 `--stats-json` を出すか、ビューアで下顎頭の Z 範囲を
  読んで決める。下顎頭だけに絞りたい場合は筋突起が混じらない高めの値にする。

### `--roi-sphere CX CY CZ R`（手動・座標既知の研究/バッチ向け）

`--roi-sphere` は **source** を中心(cx,cy,cz)・半径 r[mm] の球の内側にクリップしてから距離を
計算する。座標を数値で与えるため、3Dビューアで関節面の座標を読める研究・再現実験向け。
日常読影では上の自動 ROI を推奨。`--roi-near-target` とは同時指定不可。

- **距離は source 各点 → target 表面の最近接距離**なので、source を ROI で絞っても**残した部位の
  距離値は厳密に不変**。点数が減るぶん純粋に速くなる（間引きと違い形状の劣化もない）。
- **target は触らない。** target を切ると切断端で最近接面が消えて距離が過大化する。だから ROI は
  source 側だけにかけている（`--source-only-decimate` と同じ思想）。
- 中心座標は対象メッシュの座標系の値。だいたいの位置は `--stats-json` を一度出すか、ビューアで
  関節面の座標を読んで決める。半径は関節面が収まる程度（例: 6–10mm）から。
- ROI クロップは間引きより先に適用される（早く点を捨てるほど速い）。`--roi-sphere` と
  `--reduction` は併用可。
- 球内に source 点が1つも入らないと実行を中止する（中心・半径の指定ミス検知）。

使い分け: 全体マップが要るとき＝`--source-only-decimate --reduction`。
関節面など局所だけ精密に速く見たいとき＝`--roi-sphere`（必要なら + 間引き）。

## 座標系・整列（最重要の落とし穴）

距離は2枚のメッシュが**同一座標系に置かれていること**を前提とする。別スキャン・別モダリティ由来で
位置がずれていると、距離は「実際のクリアランス」ではなく「位置ずれ」を測ってしまう。

- 同一 CBCT / 同一スキャンから切り出した2面 → そのまま使える。
- 別スキャン（矯正前後、MR と CBCT 等）→ **先に整列**する。元アプリは SimpleITK で MR↔CBCT を
  レジストレーションしていたが、本スキルは範囲外。OpenMediCAD のランドマーク→ICP 等で揃えてから渡す。

## マルチスレッド（任意）

元アプリは VTK の SMP バックエンドを STDThread にして距離計算を並列化していた。CLI でも有効化したい
場合は実行前に環境変数や `vtk.vtkSMPTools.SetBackend("STDThread")` を足せるが、既定の単発実行でも
上記の高速化設定により実用速度は出る。まずデシメーションで調整するのが手軽。

## 出力の確認

- `--stats-json` の min が 0 付近で max が想定上限近く → 正常にクリアランス分布が取れている。
- min も max も極端に大きい → 位置ずれの疑い（整列を見直す）。
- 標準出力には距離の要約統計と点数しか出さない（メッシュ座標は展開しない）。
